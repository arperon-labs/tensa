# TENSA Writer Assistant

Activate this skill when the user is working on prose narrative inside
Studio's Storywriting Workspace, or when they ask you to draft, edit,
or critique prose anchored in a TENSA narrative. Two workflows live
here:

- **Workflow 1: Setup.** Build the canonical skeleton — narrative,
  plan, cast, pinned facts, chapter-level Arc situations. Hand off to
  Studio for prose work.
- **Workflow 2: Co-writing loop.** Stay in chat. You — the assistant —
  are the writer. TENSA is RAG, memory, continuity oracle, and quality
  gate. Draft scene-by-scene, write prose back via
  `set_situation_content`, run `check_continuity` + `run_workshop`
  between drafts, commit revisions at chapter boundaries.

Workflow 1 is default. Escalate to Workflow 2 when the user says
something like "let's draft the book here", "write Chapter 3", or
"keep going, you do the prose". Workflow 1 always runs first.

> ⚠ Synthetic narratives generated by EATH (see `/synth/*` endpoints
> + TensaQL `GENERATE NARRATIVE`) are NOT for writing. Workshop /
> pinned facts / plans / prose generation will technically run on
> synth output but produce nonsense — the underlying scenes are
> blank. Use real ingested narratives only. If a user asks to "write
> a chapter for the synthetic narrative `xxx-synth-1`", redirect
> them to a real source narrative first. (Synth is for null models,
> scaling tests, wargame substrates — see the `tensa-synth` skill.)

## Writer-specific vocabulary

- **Narrative plan** — canonical logline, synopsis, premise, themes,
  plot beats, style, length, target audience, comp titles, content
  warnings. One per narrative (`/narratives/:id/plan`). PATCH
  semantics are deep-merge — `{"style": {"pov": "first"}}` only
  touches `style.pov`; arrays and scalars replace wholesale.
- **Revision** — a snapshot commit of narrative state. Restoring a
  revision auto-commits the current state first.
- **Pinned fact** — a continuity statement ("Alice is left-handed",
  "The storm hits on Thursday") that `check_continuity` tests against
  proposed prose.
- **Cast & Places** — entities tagged with the active narrative
  (Actors → cast, Locations → places).
- **Situation = Scene** in writer terms. `situation.narrative_level`
  sorts Story → Arc → Sequence → Scene → Beat → Event. In Workflow 1,
  chapters are usually created as `Arc`-level stubs; in Workflow 2
  you create child `Scene`-level situations under each Arc and write
  their prose in `raw_content`.
- **Manuscript order** — authorial sequence (binder order).
  Distinct from `temporal.start` so narrated order can diverge from
  chronology. `manuscript_order: u32` on situation. Set it on each
  scene you create so exports and `get_scene_context` can line them
  up.
- **Workshop report** — running critique snapshot (POV, pacing,
  voice, thematic coherence). Tiered: `cheap` (deterministic only,
  instant, free), `standard` (LLM-enriched on top findings), `deep`
  (async, poll via `list_workshop_reports` / `get_workshop_report`).
- **Cost ledger** — per-narrative log of AI operation costs. Fitness-
  loop chapter generation writes entries with stable
  `kind = "chapter_gen_fitness"` and a `metadata = {iteration, score}`
  payload per attempt. `get_writer_cost_summary` aggregates.
- **Style embedding** — opaque dense voice vector trained from an
  author's corpus, stored at `se/{uuid}`. Used as a *target* for
  fitness-driven generation; rendered into the prompt via its
  `source` enum (single author / blend / genre / custom label),
  never decoded back to text.
- **Fitness loop** — generate → score → revise iteration that pushes
  generated prose toward a target `NarrativeFingerprint`. Scores the
  **prose layer only** (sentence length, dialogue ratio, FK grade,
  etc.); the structure layer is out of scope per iteration because
  re-ingesting each attempt would be prohibitive. Returns the
  **best-scoring attempt across iterations**, not the last one.
- **Calibration TODO** — the `fitness_threshold` defaults to `0.80`,
  pending a held-out same-author baseline. Treat it as a tuning
  knob, not a verdict, until calibration lands.

## Finding the REST API base URL

The TENSA REST API port varies by install (`TENSA_ADDR` env var
controls it — defaults in docs cite `3000`, but operators often run on
`4350` alongside a Studio dev server on `4300`). Don't hardcode the
port. `get_writer_workspace` returns `api_base_url` in its response
when the MCP client is talking to a remote REST server — read that
instead of probing ports.

## Default behaviour

- **Preserve voice**. When extending or editing prose, match the
  user's style — diction, sentence length, POV, tense. In Workflow 2,
  pull the plan's `style` targets at the start of every chapter and
  re-read them before each scene.
- **Flag continuity hits**. After writing a scene, *always* call
  `check_continuity`. If it returns any `conflict`-severity warning,
  revise before calling `set_situation_content`. `advisory`
  warnings are worth mentioning to the user but don't block.
- **Respect author intent over optimisation**. If the user says the
  scene should feel unresolved, don't neaten it. If the plan's style
  says "fragmentary sentences", don't smooth them out.
- **Commit revisions on big changes**. Before a destructive edit or
  a whole-chapter pass, call `commit_narrative_revision` with a
  descriptive message. It no-ops when the snapshot matches HEAD, so
  it's safe to call defensively. Use `list_narrative_revisions` to
  inspect history and `diff_narrative_revisions` to compare two
  revisions of the same narrative. `diff_narratives` is a different
  tool — it structurally diffs *two different narratives*.
- **Never invent character names or plot beats.** Pull from the
  existing plan / cast when uncertain; call `query` or `get_entity`
  to check.

## Writer-relevant tools

Beyond the base read/query/create tools, the writer surface adds:

- **`get_writer_workspace`** — always call first. Returns counts,
  plan status, recent revisions, next-step suggestions, and (via
  HttpBackend) the REST base URL.
- **`get_narrative_plan` / `upsert_narrative_plan`** — read / merge-
  update the plan. Deep-merge on nested objects so partial patches
  don't clobber sibling fields.
- **`list_pinned_facts` / `create_pinned_fact`** — continuity canon.
  Prefer over-pinning.
- **`check_continuity`** — deterministic prose validator. Free, fast,
  no LLM. Severities are `conflict | advisory`.
- **`run_workshop`** — tiered critique (`cheap | standard | deep`).
- **`list_workshop_reports` / `get_workshop_report`** — retrieve past
  critique runs and deep-tier async results.
- **`commit_narrative_revision` / `list_narrative_revisions` /
  `diff_narrative_revisions` / `restore_narrative_revision`** — git-
  like history. `restore_narrative_revision` requires `author`
  (attribution for the safety auto-commit).
- **`get_writer_cost_summary`** — aggregate the cost ledger over a
  time window (`24h`/`7d`/`30d`/`all`).
- **`set_situation_content`** (Workflow 2) — replace a situation's
  `raw_content`. Accepts either a plain string (wrapped as one Text
  block) or an array of `{content_type, content}` blocks. Optional
  `status` stamps a workflow state (`"first-draft"`, `"revised"`,
  `"final"`). Content replaces wholesale — commit a revision first
  if you want to preserve prior prose in history.
- **`get_scene_context`** (Workflow 2) — one-call context bundle:
  plan, pinned facts, optional POV character profile, current scene,
  N preceding scenes in manuscript order. Reduces per-scene calls
  from ~5 to 1. `lookback_scenes` defaults to 2, max 10.
- **`generate_chapter_with_fitness`** — optional escalation for
  Workflow 2. Submit a server-side chapter generation job with
  fingerprint targeting and SE conditioning. Args: `narrative_id`,
  `chapter`, `voice_description?`, `style_embedding_id?`,
  `target_fingerprint_source?`, `fitness_threshold?` (default 0.80),
  `max_retries?`, `synchronous?`. Returns `{job_id}`; poll with
  `job_status` / `job_result`. Use when the assistant-as-author loop
  is failing a style-fingerprint target or the user wants
  deterministic reproducibility.

## Narrative analysis toolbox

TENSA ships a narratology analysis layer (sprints D9–D11). Most tools
are deterministic and run in milliseconds — free critique that
catches structural drift a human editor would flag. Use them at the
touchpoints documented in Workflow 2; don't run them on a partial
draft and treat findings as final.

| Tool | Takes | Cost | When to run | Output / action |
|---|---|---|---|---|
| `get_health_score` | `narrative_id` | free | after every chapter | `health_score` (0–1), pathology counts by severity, worst chapter. < 0.5 = structural error; flag to user. |
| `diagnose_narrative` | `narrative_id`, `genre?` | free | after last chapter | full `DiagnosisResult` with pathologies (orphaned setups, impossible knowledge, causal orphans, motivation discontinuities, irony collapses, arc abandonment, pacing arrhythmia, …) and severities. Pass `genre` when plan has one. |
| `detect_focalization` | `narrative_id` | free | after every chapter | POV segments + switch patterns. If the plan says single focalizer and this shows drift, find the offending scene and revise via `set_situation_content`. |
| `classify_scene_sequel` | `narrative_id` | free | after every chapter | Swain/Bickham scene-sequel rhythm + pacing score. Low score = back-to-back scenes with no reflection beats, or too much sequel. |
| `compute_dramatic_irony` | `narrative_id` | free | after every chapter (mystery/thriller) | per-scene reader-vs-character knowledge gap. Gap should *evolve*, not flatline. |
| `detect_commitments` | `narrative_id` | free | midpoint + before final | Chekhov's guns: setup-payoff pairs, unpaid promises. Each unresolved commitment past the final chapter is a pathology. |
| `get_commitment_rhythm` | `narrative_id` | free | midpoint | per-chapter tension curve + fulfillment ratio. Flags chapters that make promises without paying off any. |
| `detect_character_arc` | `narrative_id`, `character_id?` | free | after ⌈N/2⌉ chapters + final | per-character arc type (positive change, flat, negative corruption, …) and transformation trajectory. Compare against plan intent. |
| `detect_subplots` | `narrative_id` | free | after ⌈N/2⌉ chapters | subplot threads via community detection on the situation graph. |
| `extract_fabula` / `extract_sjuzet` | `narrative_id` | free | on demand | fabula = chronological order; sjuzet = discourse order. Debugging tool for non-linear narratives — confirms your reordering matches intent. |
| `suggest_reordering` | `narrative_id` | free | on demand | alternative sjuzet orderings that maximize dramatic irony or commitment tension. Surface as suggestions; don't apply without user OK. |
| `compute_essentiality` | `narrative_id` | free | after last chapter (over length) | per-element 0–1 score — low = cut candidate, high = anchor. Pair with length targets to propose scenes to drop. |
| `suggest_narrative_fixes` | `narrative_id` | free (re-diagnoses if stale) | after `diagnose_narrative` | returns a list of `SuggestedFix` objects keyed to each pathology (mark-red-herring, add-commitment, move-situation, fulfill-commitment, …). **Always show each fix to the user before applying.** |
| `apply_narrative_fix` | `narrative_id`, one `SuggestedFix` | free | after user approves | deterministically mutates the hypergraph. Commit a revision first. |
| `auto_repair_narrative` | `narrative_id`, `max_severity?`, `max_iterations?` | free | rarely — only on explicit user request | loops diagnose → apply safe (confidence ≥ 0.5) fixes. **Mutates. Commit a revision first.** Prefer the `suggest_narrative_fixes` → manual approval loop. |
| `simulate_counterfactual` | `situation_id`, `intervention_target`, `new_value` | paid (INFER job) | on user ask ("what if Alice had refused?") | submits an async job; poll `job_status` / `job_result`. |
| `find_cross_narrative_patterns` | list of `narrative_ids` | paid (INFER job) | if user has a corpus of prior work | submits async; returns recurring motifs + similarity scores. |

Notes:

- All non-async tools return data in-band; no polling needed.
- Running the same deterministic tool twice is cheap — don't cache
  results across chapters, rerun them.
- Genre tuning: `diagnose_narrative(genre="thriller"|"literary_fiction"|"epic_fantasy"|"mystery"|"generic")` loads a different weighting matrix. Pull `genre` from the plan when present; default `generic`.

## Workflow 1 — Setup (always first)

Build the canonical skeleton. Zero LLM cost — pure deterministic
CRUD. Goal: a narrative that Studio's `/story/:id` dashboard renders
as "ready to draft".

1. **Orient.** `get_writer_workspace(narrative_id)` — returns counts
   + next-step suggestions + `api_base_url`.
2. **Narrative.** `create_narrative` if the slug is free.
3. **Plan.** `upsert_narrative_plan` — start with logline + synopsis
   + premise; extend with themes, central_conflict, plot_beats,
   style, length, setting, custom fields in follow-up calls. See the
   `upsert_narrative_plan` tool description for exact field shapes.
4. **Full cast — not just Actors.** `create_entity` with
   `narrative_id` set for:
   - **Actor** — every named character (protagonist, antagonists,
     confidants, mentors, background names that recur).
   - **Location** — every recurring setting (town, building, room,
     landmark). Store geo in `properties.spatial` when known so the
     Studio Geo Map populates.
   - **Object** — MacGuffin, weapon, letter, ledger, heirloom. Anything
     that changes hands, gets searched for, or carries symbolic weight.
   - **Organization** — shops, families, factions, agencies, clubs.
   - **Concept** — abstractions the plot turns on (reputation, a
     secret, a rumour, a legal doctrine). Skip for pure character
     studies; over-use dilutes the graph.
   Omitting entity kinds kills downstream analysis: Location drives
   geo/map views and proximity inference, Object drives commitment
   tracking ("whose hands has the ledger been in?"), Organization
   drives faction/community detection.
5. **Pinned facts.** `create_pinned_fact` — each canonical fact. Use
   `entity_id` for character-scoped facts, omit for narrative-wide.
   Over-pin; each fact is `check_continuity`'s unit of enforcement.
6. **Chapter situations.** `create_situation` per chapter with
   **`name` set to the chapter title** (e.g. "Chapter 1 — The
   Inventory"), `narrative_level: "Arc"`, `manuscript_order` set, a
   beat summary in `raw_content`. Set `discourse` at creation —
   focalization UUID + voice — or you'll be back-filling later.
   Scenes without `name` render as `"(untitled)"` across the Studio
   outline, manuscript header, workshop findings, and compile output —
   *always* pass `name`, don't rely on the reader finding the title
   inside `raw_content`.
7. **Verify.** `get_writer_workspace` again. Expected minimums before
   moving on: ≥ 1 Actor per plot_beat focalizer, ≥ 1 Location per
   recurring setting in the plan, ≥ 1 Arc-level situation per chapter.
   If the graph looks sparse, stop and ask — a thin graph ships a thin
   Workshop.

At this point hand off to Studio (`navigate_to /story/:id/plan` or
`/story/:id/generate`) **or** escalate to Workflow 2 if the user
wants to draft in chat.

## Workflow 2 — Co-writing loop (assistant-as-author)

The user keeps the session open; you draft the book. TENSA's role:
memory, retrieval, continuity enforcement, version control, critique.
Your role: creative writing, decision-making, pacing.

### Loop structure

```
for chapter_idx, chapter_arc in enumerate(plan.plot_beats):
    ctx := get_scene_context(narrative_id, situation_id=<chapter_arc uuid>,
                              pov_entity_id=<focalizer>, lookback_scenes=2)
    plan scene breakdown (3-5 Scene children under the Arc) — each
    scene needs: title, POV, participants (actor + location + any
    objects), primary action, intended beat function (setup / try /
    reversal / payoff), and a game-theoretic note if applicable
    (PrisonersDilemma, Coordination, ZeroSum, AsymmetricInformation,
    StagHunt, Ultimatum, Chicken) so downstream game-theory engines
    have data.

    for each scene:
        # 1. The situation itself.
        scene_id := create_situation(
            name=<scene title>,             # REQUIRED — outline shows "(untitled)" otherwise
            narrative_level=Scene,
            parent_situation_id=<chapter_arc uuid>,
            manuscript_order=<seq>,
            raw_content=<beat-sketch>,
            discourse={focalization, voice},
            spatial=<Location lat/lng or place name>,      # drives geo map
            game_structure=<GameStructure JSON if strategic>,  # drives game-theoretic infer
            narrative_id=<nid>)

        # 2. Participations — the hypergraph is edges, not just nodes.
        # A scene with no add_participant calls is invisible to
        # dramatic irony, commitments, mean-field games, subplot
        # detection, and every relationship-based query. Over-link;
        # omission is the expensive mistake.
        for each participant in scene:
            add_participant(
                entity_id=<actor/location/object uuid>,
                situation_id=scene_id,
                role=<Protagonist|Antagonist|Witness|Target|Confidant|
                      Informant|Instrument|Recipient|Bystander|
                      SubjectOfDiscussion|{"Custom":"Setting"}>,
                action=<"confesses"|"arrives"|"hides X"|...>,
                info_set={knows_before: [...], learns: [...],
                          reveals: [...]})           # REQUIRED for
                                                     # dramatic irony /
                                                     # belief reasoning
            # Location participants use role Custom:"Setting".

        # 3. Causal wiring. Emit a causal link from the scene that
        # *causes* this one — usually the previous sibling scene, but
        # cross-chapter causation is common (reveal → flashback cause,
        # confession → prior lie). Without these edges, narrative
        # diameter, causal influence, and SCM counterfactual analyses
        # all degenerate. The REST path is POST /causal-links with
        # {from_situation, to_situation, causal_type, mechanism,
        # strength}; no MCP tool yet — call via the api_base_url.

        ctx := get_scene_context(narrative_id, situation_id=scene_id,
                                  pov_entity_id=<focalizer>, lookback_scenes=2)

        # 4. Draft the prose, check continuity, write back.
        draft full prose (your writing, conditioned on ctx)
        warnings := check_continuity(narrative_id, prose)
        if any warning.severity == "conflict":
            revise prose to resolve; loop until clean
        set_situation_content(scene_id, prose, status="first-draft")

        # 5. Fire-and-forget analytical hooks per scene. Each of these
        # is cheap, deterministic, and their outputs persist to the
        # an/* KV prefixes so Studio views and Workshop pick them up
        # immediately. Skip only if the scene is pure atmosphere with
        # no strategic/informational content.
        if scene had a strategic/bargaining beat:
            submit job: CausalInference | GameTheory | CognitiveHierarchy
            (via POST /jobs, or the dedicated MCP tools when present)
        if scene changed belief state (reveal/deceit):
            compute_dramatic_irony(narrative_id, chapter_id=chapter_arc)
        if scene introduced a promise/threat/question/puzzle:
            detect_commitments(narrative_id)  # deltas roll into
                                              # commitment rhythm

    # Per-chapter post-flight (see below)
    post_flight(narrative_id, chapter_arc)

    # Midpoint structural audit (see below)
    if chapter_idx + 1 == ceil(len(plot_beats) / 2):
        midpoint_audit(narrative_id)

    commit_narrative_revision(narrative_id, message=f"Ch {chapter_idx+1} first draft",
                              author="assistant")

# After last chapter
final_pass(narrative_id, genre=plan.genre)

# 6. Embeddings + global analyses. Scenes written via
# set_situation_content don't auto-embed — trigger the backfill once at
# the end so vector search, NEAR queries, community detection, and the
# Studio Embedding Space all work. REST: POST /embeddings/backfill
# (body: {narrative_id}).
embed: call POST /embeddings/backfill

# Optional one-shot repairs (idempotent, safe to run twice):
# POST /narratives/:id/causal-links/backfill-adjacent  — synthesise
#   weak sequential edges where you forgot to emit causal links.
# POST /narratives/:id/names/backfill-from-content     — rescue any
#   scenes that slipped through with name=None.

commit_narrative_revision(narrative_id, message="First draft complete",
                          author="assistant")
```

### Per-chapter post-flight

Run after every chapter commit. All tools are free, deterministic, in-
band. If any surface a real issue, propose a targeted re-draft of the
offending scene via `set_situation_content` and re-commit.

```
post_flight(narrative_id, chapter_arc):
    report := run_workshop(narrative_id, tier="cheap")
    health := get_health_score(narrative_id)        # quick pulse

    focal  := detect_focalization(narrative_id)
    if plan.style.pov is "single focalizer" and focal shows switches:
        → identify offending scene from focal, revise via set_situation_content

    pacing := classify_scene_sequel(narrative_id)
    if pacing.score < threshold or back-to-back scenes with no sequel:
        → flag to user; propose inserting a reflection beat

    if plan.genre in {"mystery", "thriller", "crime"}:
        irony := compute_dramatic_irony(narrative_id)
        if reader-vs-character gap is flat across the chapter:
            → the chapter didn't advance information state; flag

    if report has >2 findings worth LLM enrichment:
        run_workshop(narrative_id, tier="standard",
                     focuses=[pacing, continuity, characterization])
```

### Midpoint audit

Run once, at the chapter where half the plan is drafted. Surface
promises made vs. payoffs still owed, and confirm subplots haven't
drifted from the plan.

```
midpoint_audit(narrative_id):
    commits := detect_commitments(narrative_id)
    rhythm  := get_commitment_rhythm(narrative_id)

    # Each unresolved commitment is something the remaining chapters
    # must either pay off, mark as red-herring, or cut. Tell the user
    # the list explicitly — do not let them float.
    report unpaid commitments to user

    subplots := detect_subplots(narrative_id)
    if subplot count > plan.expected_subplots + 1:
        → threads have sprawled; propose merge or cut
    if subplot count < plan.expected_subplots - 1:
        → threads haven't emerged; check the plan is realistic

    arcs := detect_character_arc(narrative_id)  # all characters
    for each lead in plan.cast:
        if detected arc type ≠ plan intent:
            → flag to user; plan may need updating via upsert_narrative_plan

    # If plan drift is real, patch it before drafting the back half.
    if any of the above surface a real mismatch:
        confirm with user, then upsert_narrative_plan with corrections
```

### Final-pass deep analysis

Run once, after the last chapter is committed. This is where the
expensive tools earn their keep — the narrative is complete enough
for their signal to be trustworthy.

```
final_pass(narrative_id, genre):
    diag := diagnose_narrative(narrative_id, genre)
    health := get_health_score(narrative_id)
    show user: health_score, worst chapter, pathology count by severity

    if diag.pathologies is non-empty:
        fixes := suggest_narrative_fixes(narrative_id)
        for each fix:
            show user the pathology and the proposed mutation
            if user approves:
                commit_narrative_revision(message="Before applying fix: <desc>")
                apply_narrative_fix(narrative_id, fix)
            else:
                skip

    if total_words > plan.length.max_words:
        essent := compute_essentiality(narrative_id)
        surface lowest-essentiality scenes as cut candidates
        # do not cut autonomously — propose to user

    # Non-linear structure check (only if plan.custom includes
    # narrated-order guidance)
    if user wants to audit narrated vs. chronological order:
        fabula := extract_fabula(narrative_id)
        sjuzet := extract_sjuzet(narrative_id)
        diff   := suggest_reordering(narrative_id)
        present any high-value reorderings as proposals

    if user requested whole-book LLM critique:
        run_workshop(narrative_id, tier="standard")

    final health := get_health_score(narrative_id)
    report health trend (midpoint → final)
```

### Context-budget discipline

A full novel doesn't fit in a prompt. Per scene, your context should be:

- **Always:** plan (logline, synopsis, style, length targets, themes)
  and pinned facts for characters in this scene. One
  `get_scene_context` call gives you these.
- **Usually:** the last 2 scenes' prose (`preceding_scenes` in the
  bundle, default `lookback_scenes=2`).
- **When needed:** for callbacks to earlier events, use
  `ask "What did X say about Y in chapter 2?" MODE local OVER "<nid>"`
  — RAG retrieves targeted context without bloating the prompt.
- **When structural:** `query` for temporal slicing
  (`MATCH (s:Situation) WHERE s.narrative_id = "..." AND s.temporal.start <
  "..." RETURN s`).
- **Don't load every prior scene into a prompt**. That's what RAG
  (`ask`) and the bundle (`get_scene_context`) are for.

### When to call `check_continuity`

Every time you draft prose, before `set_situation_content`. It's
deterministic and free. Failure modes:

- `conflict` severity → silent contradiction with a pinned fact.
  Revise and retry.
- `advisory` severity → borderline; mention to the user, let them
  choose.

### When to call `run_workshop`

- `cheap` after every chapter — instant, deterministic, free. Catches
  pacing anomalies, stub characters, structural holes.
- `standard` at chapter end if `cheap` surfaced >2 findings, or at
  manuscript end before the final commit. LLM-enriched top findings.
- `deep` only on user request; it's async and paid.

### When to commit a revision

- At the end of every chapter (first-draft commits).
- Before a whole-book revision pass.
- Before `restore_narrative_revision` (it auto-commits anyway — this
  is belt-and-suspenders).
- Before any destructive `set_situation_content` that rewrites
  already-committed prose.

### When to escalate to `generate_chapter_with_fitness`

Use the server-side generator when:

- The user has a style-fingerprint target (comp title, voice profile)
  and asks you to match it numerically.
- The user wants reproducibility — the server-side job is logged in
  the cost ledger with `kind = "chapter_gen_fitness"` and a scored
  fitness trajectory.
- Your own attempts at a scene keep drifting off voice and you want
  the best-of-k iteration.

Otherwise, stay in the chat-native loop — it's cheaper, interactive,
and the user can steer every scene.

## Typical intents

- "Add a scene where Alice confronts Bob" → look up cast + plan via
  `get_scene_context`, propose situation text, `create_situation`,
  draft prose, `check_continuity`, `set_situation_content`, optional
  commit.
- "Is this line consistent with the timeline?" → `check_continuity`
  on the line; if clean, check pinned facts + nearby situations via
  `get_scene_context`.
- "Polish this paragraph" → edit in the user's voice, return the
  prose, `check_continuity`, then `set_situation_content` after
  confirmation.
- "What should happen next?" → consult the narrative plan + recent
  beats via `get_scene_context`, propose 2–3 options, let the user
  pick before drafting.
- "Keep going, write Chapter 3" → engage Workflow 2 scene-by-scene.
- "Rewrite Chapter 2 in first-person past tense" → commit a revision
  first, upsert the plan's `style.pov`/`style.tense`, walk scenes
  calling `set_situation_content` with updated prose, commit again.

## Don't

- Silently rewrite whole chapters without committing a revision
  first. Offer paragraph- or scene-scoped edits the user can accept
  or reject.
- Silently change pinned facts. If the narrative evolves, tell the
  user and update the pinned fact explicitly.
- Skip `check_continuity` before `set_situation_content`. It's free
  and fast.
- Load every prior scene into the prompt. Use RAG (`ask`) or the
  bundle (`get_scene_context`) instead.
- Use "voice" tools that don't exist. Workflow 2 uses
  `set_situation_content` to commit prose you wrote; there is no
  prose-transform tool for retroactive voice rewriting — do the
  rewrite yourself in chat.
- Treat the server-side `generate_chapter_with_fitness` as a
  replacement for the chat-native loop. It's a specialized escalation
  for fingerprint-matching, not the default.
- Call `auto_repair_narrative` without explicit user approval. It
  mutates the hypergraph, even with conservative defaults. Prefer
  `suggest_narrative_fixes` → show each fix → `apply_narrative_fix`
  after approval. Commit a revision before either path.
- Apply `suggest_narrative_fixes` in a loop without surfacing each
  fix. Each fix is a structural edit the user needs to see — no
  "auto-accept all".
- Run `diagnose_narrative` on a partial draft and treat the pathology
  report as final. Many pathologies (orphaned commitments, arc
  abandonment) resolve when the last chapter lands — run the full
  diagnosis as part of `final_pass`, not mid-draft.
- Run structural analysis (`detect_focalization`,
  `detect_commitments`, `detect_character_arc`, `detect_subplots`)
  immediately after setup. These tools need real scene prose for
  signal; on a stub-only narrative they return noise.
- Submit `simulate_counterfactual` or `find_cross_narrative_patterns`
  without the user asking. Both are paid INFER jobs with
  asynchronous polling and are almost never essential to a draft.

## Bonus: opinion-dynamics scoring (Phase 16c)

Opinion dynamics CAN actually be useful for writers — unlike the
synthetic-generation surface, which produces actor data without
manuscript prose. Each character's "opinion" can be initialized from
extracted attitudes, then run BCM on the participation hypergraph to
score whether a plot's character-opinion trajectory is
**consensus-forming** or **polarization-forming**.

```sql
INFER OPINION_DYNAMICS( confidence_bound := 0.3, variant := 'pairwise' )
    FOR "novel-1"
-- → report.num_clusters: 1 = ensemble unifies; > 1 = factionalisation.
-- → report.polarization_index: closer to 1 = bimodal final state.
```

For deeper opinion-dynamics workflows (phase-transition sweeps, echo
chamber analysis, wargame OpinionShift rewards), switch to the
`tensa-opinion-dynamics` skill bundle. Idiom links:
- "Does this narrative converge or fragment?"
- "What confidence threshold makes this group find consensus?"
- "Are there echo chambers in this corpus?"

## Bonus: uncertain-fact modelling (Fuzzy Sprint, v0.78.0)

When a plot hinges on graded / partial facts (a rumour that "most
villagers suspect the baron", an alliance that holds "somewhat" but
not fully), the `tensa-fuzzy` skill bundle exposes graded Peterson
syllogism verification, intermediate quantifiers ("most / many / few"
ramps), and Mamdani fuzzy rules over character properties — useful
for validating plot logic without forcing every premise into a
crisp boolean.
